Introduction

Report runner service is a microservice for generating BIRT reports.

Overview

We attempt to adhere to REST standards although running a report is really an action and not a resource. However we use POST for report run actions because it can be thought of as adding a new request, where a request is a resource. We use GET for retrieving reports and report status.

When the payload is the generated report, it is returned with the appropriate MIME type and a response code of 200. If there was an error generating the report, a response code of 500 (internal server error) is returned. If the report design file was not found, 404 (not found) is returned. If the output format is not recognized, 406 (not acceptable) is returned.

The server can be configured to use the open-source BIRT libraries or the OpenText commercial BIRT libraries.

Requests

Report run request

This request is used to immediately generate a report and return the output in the response.

request fields:

Path Type Description

designFile

String

The full path to the BIRT design file on the server file system

format

String

The report output format: HTML, PDF, XLS, or any other format supported by the BIRT engine

runThenRender

Boolean

Whether to build with separate run and render phases

parameters

Object

The parameters in the form {"name": value, …​}, where value may be a string, number or boolean for single value parameters or an array of string, number, or boolean for multi-valued parameters.

securityToken

String

The security token for this request. This is required only if database security has been enabled in the configuration.

Examples:

curl:

$ curl 'http://localhost:8080/run' -i -X POST \
    -H 'Accept: application/json' \
    -H 'Content-Type: application/json' \
    -d '{
	"designFile": "/disk1/home/innovent/projects/BIRT-Report-Runner-REST-Service/src/test/resources/com/innoventsolutions/birt/test/test.rptdesign",
	"format": "pdf",
	"runThenRender": true,
	"parameters": {
		"keyFilter": "a.*"
	}
}'

http request:

POST /run HTTP/1.1
Accept: application/json
Content-Type: application/json
Content-Length: 232
Host: localhost:8080

{
	"designFile": "/disk1/home/innovent/projects/BIRT-Report-Runner-REST-Service/src/test/resources/com/innoventsolutions/birt/test/test.rptdesign",
	"format": "pdf",
	"runThenRender": true,

		"keyFilter": "a.*"

Report status request

This request is used to get the status of a running or finished job.

Request path parameters: ./status/{uuid}

Parameter Description

uuid

The identifier returned from submit

Response fields:

Path Type Description

reportRun

Object

The report run request

email

Object

The email request

startTime

String

The time when the report generation started

finishTime

Null

The time when report generation and email finished or null if not finished

reportFinishTime

Null

The time when the report generation finished or null if it is still running

duration

Number

The number of milliseconds it took to generate the report

finished

Boolean

Whether the report generation is finished

errors

Array

List of exceptions that were encountered during report generation

emailErrors

Object

List of exceptions that were encountered during sending email

Examples:

curl:

$ curl 'http://localhost:8080/status/b92b8e8f-4b75-41cc-9fcc-9dec48812266' -i -X GET \
    -H 'Accept: application/json'

http request:

GET /status/b92b8e8f-4b75-41cc-9fcc-9dec48812266 HTTP/1.1
Accept: application/json
Host: localhost:8080

Parameters

In the run and submit requests, parameters are specified in JSON as an object in the form {"name": value, …​}, where name is a parameter name and value may be a string, number, boolean, or object. For example: {"a-number": 1, "a-string": "apples", "a-boolean": true}.

Objects are used to specify date or time data types. If an object is used it must have two fields: "type" and "value". Type must be one of "date", "datetime", or "time", and value must be a string. The format for dates is "yyyy-MM-dd", for datetime "yyyy-MM-dd HH:mm:ss", and for time "HH:mm:ss". For example: {"a-date": {type: "date", value: "2016-06-21"}}.

Additionally for multi-value parameters you can specify an array of values, like this: {"name": [value, …​], …​}. For example: {"some-numbers": [1, 2, 3]}. The same types as single-value parameters can be used.

Building

To build the jar from source, change directory to the root folder (the folder containing pom.xml) and type

mvn package

The build will run a series of unit tests and also build the documentation you are reading.

Runtime Configuration

This server is configured by a java properties file. The full path to the properties file must be specified in an environment variable named REPORT_RUNNER_PROPERTIES prior to executing the jar file. The following properies may be specified.

Note: in the following, ~ refers to your home directory.

birt.runner.outputDir

The default location of output files. If not absolute it will be relative to ~. The default is ~/reportRunnerTest/output.

birt.runner.workspace

The default location of report design files. If not absolute it will be relative to ~. The default is ~/reportRunnerTest/reports.

birt.runner.runtime

For versions of BIRT less than 3.7 and the OpenText commercial version, this should be the full path to the directory containing the eclipse plugin directory. In the case of open source birt-runtime it should be the ReportEngine directory. For OpenText commercial it can be iServer/Jar/BIRT/platform in the iServer installation directory or WEB-INF/platform in the OJC webapp. For open source versions of BIRT greater than or equal to 3.7 it should be omitted.

birt.runner.resources

The location of resources needed for reports, such as .rptlibrary files, javascript files, css files, and jar files. If not absolute it will be relative to the location of this properties file. The default is the resources directory in the same directory as this properties file.

birt.runner.scriptlib

The location of jar files needed for report event handlers. If not absolute it will be relative to the location of this properties file. The default is the lib directory in the same directory as this properties file.

birt.runner.reportFormat

The default format for all reports. The default is PDF.

birt.runner.baseImageURL

The base URL used for images in HTML reports. If absent, no base image URL will be provided.

birt.runner.logging.properties

The logging properties file used by the BIRT report engine. If not absolute it will be relative to the location of this properties file. The default is "logging.properties" in the same directory as this properties file.

birt.runner.logging.dir

The filesystem path to the directory for log files produced by the BIRT report engine. If not absolute it will be relative to the location of this properties file. The default is the log directory in the same directory as this properties file.

birt.runner.db.driver

The JDBC driver class name for database request security. If missing, no database request security will be done.

birt.runner.db.url

The JDBC URL for database request security. If missing, no database request security will be done.

birt.runner.db.username

The username required for database authentication.

birt.runner.db.password

The password required for database authentication.

birt.runner.db.query

The query for database request security. The query must accept a single parameter denoted by a question mark in which the security token will be passed and return the full report design file path/name and the timestamp of the request time in that order.

Example: select design_file, request_time from reports where security_token = ?;
birt.runner.db.timeout

The maximum number of milliseconds allowed between the timestamp returned by the database query (birt.runner.db.query) and the actual time the report request is submitted. This should be very short. The default is 5000. The request will not be allowed if the request submission time is later than the timestamp in the database plus the timeout.

birt.runner.unsecuredDesignFilePattern

This is an optional regular expression that is matched against the full design file path/names. If it matches, the request is allowed with no further security checks. If it is missing then other security measures, if any, will be enforced for all reports.

birt.runner.mail.username

The user name required for SMTP server authentication.

birt.runner.mail.password

The password required for SMTP server authentication.

birt.runner.mail.properties

The javamail properties file. If not absolute it will be relative to the location of this properties file. The default is "smtp.properties" in the same directory as this properties file. If this file does not exist, no emails will be sent.

birt.runner.mail.to

A comma separated list of email addresses. All addresses in this list will receive emails in addition to any recipients specified in the report request.

birt.runner.mail.cc

A comma separated list of email addresses. All addresses in this list will receive emails in addition to any copy recipients specified in the report request.

birt.runner.mail.bcc

A comma separated list of email addresses. All addresses in this list will receive emails in addition to any copy recipients specified in the report request.

birt.runner.mail.from

A single email address. All emails will be from this address.

birt.runner.mail.success

Whether to send emails for successful reports. The default is false. This can be overridden by a submit request.

birt.runner.mail.failure

Whether to send emails for failed reports. The default is false. This can be overridden by a submit request.

birt.runner.mail.subject.success

The default email subject for successful generations. Success subjects specified in submit requests will override this one. The default is "Success".

birt.runner.mail.subject.failure

The default email subject for failed generations. Failed generation subjects specified in submit requests will override this one. The default is "Failure".

birt.runner.mail.body.success

The default email body for successful generations. Successful generation bodies specified in submit requests will override this one. The body text will be interpreted as HTML if the birt.runner.mail.html property is "true". The body text may contain one or more of the following substitution strings:

${designFileName}

The BIRT report design filename specified in the report run or submit request.

${nameForHumans}

The human-friendly name for the report specified in the report run or submit request.

${startTime}

The date and time the report generation started.

${finishTime}

The date and time the report generation finished.

birt.runner.mail.body.failure

The default email body for failed report generations. Failed generation bodies specified in submit requests will override this one. Rules for the birt.runner.mail.body.success property are also applicable to this one.

birt.runner.mail.attachReport

True if the report output file should be attached to the email. The default is true. Submit requests can override this value.

birt.runner.mail.html

True if the email body is HTML. False if it is plain text. The default is true. Submit requests can override this value.

birt.runner.threadCount

The number of reports that can be simultaneously generated. The default is 1.

birt.runner.isActuate

True if the Actuate commercial jar files are to be used. The default is false.

Security

Unsecured Reports

A subset of reports can be unsecured by specifying the birt.runner.unsecuredDesignFilePattern property. This is a regular expression that is matched against the full report design path/name in the request. If it matches, the request is allowed without any further security checks. If it is not specified then other security constraints (if any) are applied to all reports.

Database Secured Requests

A SQL database server can be used as a back channel to secure requests. To enable this, specify all the "db" properties in the configuration. Then for each request you will need to create a random security token (UUID.getRandomUUID() is a good way to do this). Then you’ll need to insert a row into a table prior to making a /run or /submit request. The table should contain the security token, the full path/name of the report design file, and the current timestamp. The security token must also be included in the /run or /submit request. The configuration will provide a query that can retrieve these values from the database. If the security token is not found or the report request name doesn’t match the request or the request is received later than a timeout specified in the configuration, then the request will be rejected.

Configuring for OpenText libraries

If you have purchased the OpenText commercial iHub, iServer or Java Components products you can build this REST server to use the commercial libraries. There are two directories you will need to locate: lib and platform. In the iServer installation they will be in iServer/Jar/BIRT. In the case of OJC, they are in WEB-INF in the OJC webapp. The jars in lib are used during the maven build and the jars in platform/plugins are used at runtime.

  1. Use the bash script mvn-install-birt-jars to install the commercial jars into your local maven repository. For example, assuming a standard Actuate 11 iServer installation:

    ./mvn-install-birt-jars "/c/Program Files (x86)/Actuate11sp6/iServer/Jar/BIRT/lib"

    This works from the git-bash shell on windows, available from https://git-scm.com/downloads.

  2. Rename pom.xml to pom-os.xml and pom-actuate.xml to pom.xml.

  3. Build with

    mvn package

  4. Make sure your runtime properties file is correctly set up for the commercial jars:

    1. set birt.runner.isActuate to true.

    2. set birt.runner.runtime to the platform directory. For example:

      birt.runner.isActuate=true
birt.runner.runtime=C:/Program Files (x86)/Actuate11sp6/iServer/Jar/BIRT/platform

Note that if you use backslashes in a java properties file they must be doubled, like this: C:\\Program Files (x86)\\Actuate11sp6\\iServer\\Jar\\BIRT\\platform. Single forward slashes can also be used. On Windows they will be automatically converted to backslashes.

Startup

To start the server, first make sure you have defined the REPORT_RUNNER_PROPERTIES environment variable. then change directory to the target folder under the root folder (the folder containing pom.xml) and type:

java -jar BIRT-Report-Runner-REST-Service-0.0.1-SNAPSHOT.jar
